Developing a Model

Student correctly identifies whether the hypothetical model successfully captures the variation of the target variable based on the model’s R^2 score.
The performance metric is correctly implemented in code.

Agreed, this model has a good fit on our data. Wikipedia provides a nice explanation for how we calculate R^2.

In short, we're able to represent the percent of variance within the data that our model is capable of explaining.

Another way of looking at this is by comparing our model to a naive predictor. A simple model for a dataset might be to always predict the mean value. The squared error of this model would correspond with the total sum of squares. Meanwhile, our model would have a squared error corresponding with the residual sum of squares. And thus, by comparing the residual sum of squares to the total sum of squares we are determining how similar our model is to a naive predictor. If we had a negative R^2 value, this would mean our model does worse than a naive predictor (always predicting the mean value).

Correct! By splitting our data into training and testing datasets, we can more effectively evaluate our model's ability to generalize to real-world examples.

Evaluating Model Performance

Student correctly describes the grid search technique and how it can be applied to a learning algorithm.

Correct!

Since our hyperparameters can’t be directly learned, we must perform multiple experiments trying different hyperparameter values to “search” the range of possibilities for the best hyperparameter values.

Grid search is an exhaustive parameter search (trying all of the combinations) and can take a long time for models with a large number of hyperparameters. For each hyperparameter combination, we’ll evaluate the model according to our evaluative metric. The combination with the highest validation score is selected as the final model.

Random search and Bayesian optimization are two other methods that can perform better for searching high-dimensional parameter-spaces. If you want to learn more, a former Udacity student wrote an article about hyperparameter optimization techniques. For an even more advanced hyperparameter tuning technique, check out DeepMind’s blog post on population based training. Essentially, it allows your different experiments communicate and work together in finding the optimal values rather than searching in isolation.

Student correctly describes the k-fold cross-validation technique and discusses the benefits of its application when used with grid search when optimizing a model.

Then we train the model k times each time using different bucket as our testing set, and the remaining points as our training set.

It’s a small distinction, but we actually use k-1 folds for training and the remaining fold for validation. Testing is commonly a reference to the final evaluation after we’ve built our model. K-folds cross validation only looks at the training data, and further splits it into K subsets for both tuning the model parameters (using the K-1 folds for training) and find the right hyperparameter values (using the remaining fold for validation).

I know may be confusing (since the notebook instructions mention training and testing). However, if you read the sklearn documentation also referenced in the instructions this distinction is made more clearly.

Attention

If you look at the GridSearchCV documentation, we technically need to specify the scoring_fnc as a keyword argument (ie. scoring = scoring_fnc). Although this code will still run because the order is right in this case, it's strongly recommended you supply the keyword when specifying a keyword argument.

The documentation states that the GridSearchCV accepts the following arguments: 

class sklearn.model_selection.GridSearchCV(estimator, param_grid, scoring=None, fit_params=None, n_jobs=1, iid=True, refit=True, cv=None, verbose=0, pre_dispatch=‘2*n_jobs’, error_score=’raise’, return_train_score=’warn’)

The first two arguments, estimator and param_grid, are positional arguments and do not need to be addressed by name. However, they must be provided and in the correct order. The keyword arguments have default values and do not need to be called in a specific order. If you pass an argument without the keyword, it will be assumed to follow the order of arguments provided in the documentation, but it's best practice to also address the keyword arguments by name.
